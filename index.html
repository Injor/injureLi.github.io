服务发现

Posted on 2020-06-11 Edited on 2020-06-30 In 微服务

服务发现

服务发现有三个角色，服务提供者、服务消费者和服务中介，消费者调用 REST API 或者 Thrift API 的服务，需要知道服务实例的网络位置（IP和端口）

代理模式

关于服务发现和负载均衡，业界探索和总结出一些常用的模式，这些模式的核心是代理，以及代理在架构中所处的位置。

集中式代理

硬件负载均衡器F5和软件负载均衡器Nginx，在DNS域名服务器配合下实现服务发现，建立服务域名和IP地址之间的映射关系

  • 运维简单、集中治理、语言无关
  • 配置麻烦、单点问题、多一跳有性能开销
  • 大中小公司都适用，可以进行改造自注册

客户端嵌入式代理

代理以客户端形式嵌入在程序中，服务实例自注册到注册中心并定期健康检查，客户端代理自发现服务并进行负载均衡

  • 无单点问题，性能好
  • 客户端复杂、多语言麻烦、治理松散
  • Eureka、Dubbo

主机独立进程代理

代理作为独立进程部署在每一个主机上，既不是独立集中部署，也不嵌入客户应用程序，一个主机上多个消费者共用这个代理，实现服务发现和负载均衡

  • 部署运维复杂
  • Service Mesh、Istio

Eureka

Eureka is a REST (Representational State Transfer) based service that is primarily used in the AWS cloud for locating services for the purpose of load balancing and failover of middle-tier servers. We call this service, the Eureka Server. Eureka also comes with a Java-based client component,the Eureka Client, which makes interactions with the service much easier. The client also has a built-in load balancer that does basic round-robin load balancing.

  • Service Provider会向Eureka Server做Register（服务注册）、Renew（服务续约）、Cancel（服务下线）等操作。
  • Eureka Server之间会做注册服务的同步，从而保证状态一致
  • Service Consumer会向Eureka Server获取注册服务列表，并消费服务

Register

Register（服务注册），注册的服务列表保存在一个嵌套的hash map中：

  • 第一层hash map的key是app name，也就是应用名字
  • 第二层hash map的key是instance name，也就是实例名字

  1. ApplicationResource类接收Http服务请求，调用PeerAwareInstanceRegistryImpl的register方法
  2. PeerAwareInstanceRegistryImpl完成服务注册后，调用replicateToPeers向其它Eureka Server节点（Peer）做状态同步（异步操作）

Renew

Renew（服务续约）操作由Service Provider定期调用，类似于heartbeat。主要是用来告诉Eureka Server Service Provider还活着，避免服务被剔除掉。

  • instance.leaseRenewalIntervalInSeconds (Renew频率。默认是30秒)
  • instance.leaseExpirationDurationInSeconds (服务失效时间。默认是90秒)

Cancel

Cancel（服务下线）一般在Service Provider shut down的时候调用,通过对方法标记@PreDestroy，从而在服务shut down的时候会被触发，把自身的服务从Eureka Server中删除，以防客户端调用不存在的服务。

Fetch Registries

  • Service Consumer在启动时会从Eureka Server获取所有服务列表，并在本地缓存。
  • 由于在本地有一份缓存，所以需要定期更新，定期更新频率可以通过eureka.client.registryFetchIntervalSeconds配置。

Eviction

Eviction（失效服务剔除）用来定期（默认为每60秒）在Eureka Server检测失效的服务，检测标准就是超过一定时间没有Renew的服务。

  • eureka.instance.leaseExpirationDurationInSeconds (默认失效时间为90秒)
  • eureka.server.evictionIntervalTimerInMs (定期,默认为每60秒)

自保护

Eureka在运行期间会统计心跳失败的比例，在15分钟内是否低于85%,如果出现了低于的情况，Eureka Server会将当前的实例注册信息保护起来，同时提示一个警告，一旦进入保护模式，Eureka Server将会尝试保护其服务注册表中的信息，不再删除服务注册表中的数据。也就是不会注销任何微服务。

CAP

Peer之间的状态是采用异步的方式同步的，所以不保证节点间的状态一定是一致的，不过基本能保证最终状态是一致的。
按照CAP理论，Eureka的选择就是放弃C，选择AP。

Nacos

Nacos is committed to help you discover, configure, and manage your microservices. It provides a set of simple and useful features enabling you to realize dynamic service discovery, service configuration, service metadata and traffic management.

Nacos makes it easier and faster to construct, deliver and manage your microservices platform. It is the infrastructure that supports a service-centered modern application architecture with a microservices or cloud-native approach.

Service Mesh

Service Mesh又称为服务网格,每个主机上同时运行了业务逻辑代码和代理。

这个代理被形象地称之为SideCar，服务之间通过SideCar发现和调用目标服务，从而形成服务之间的一种网络状依赖关系。

然后通过独立部署的一种称之为控制平面（ControlPlane）的独立组件来集中配置这种依赖调用关系以及进行路由流量调拨等操作

RPC远程调用

Posted on 2020-06-10 Edited on 2020-06-30 In 微服务

要解决的问题

  • 协议约定问题
    • 如何规定远程调用的语法？
    • 参数如何传递，什么时候是头，什么时候是尾？
    • 如何表示数据？
  • 传输协议问题
    • 发生错误、重传、丢包、性能问题怎么办？
  • 服务发现问题
    • 一个服务实现了哪些远程调用，从哪个端口可以访问？

协议类型

文本类 二进制类
SOAP Hessian2
RESTful Protocol Buffers
  • 基于XML的SOAP协议（Simple Object Access Protocol）全称简单对象访问协议，设计是面向动作的，因而往往因为架构问题导致并发量上不去。
  • 基于JSON的RESTful协议（Representational State Transfer）表述性状态转移，是一种架构模式，主要面向资源，提供无状态服务，有利于横向扩展应对高并发。
  • 基于二进制的RPC协议Hessian2，不需要定义协议文件，是自描述的
  • 基于二进制的GRPC协议Protocol Buffers，需要写协议文件。网络传输使用 HTTP 2.0，服务治理可以使用基于 Envoy 的 Service Mesh。

Dubbo中的RPC调用

SpringCloud中的OpenFeign调用

二者对比

Dubbo

  • Dubbo中默认的RPC协议是Hessian2。为了保证传输的效率，Hessian2 将远程调用序列化为二进制进行传输，并且可以进行一定的压缩。
  • Hessian2是自描述的，不需要再定义协议文件
  • Dubbo使用Netty的网络传输框架，基于Socket实现高性能服务端
  • 还提供其他高性能序列化方式，例如Dubbox 从 Spark 那里借鉴 Kryo
  • 需要客户端和服务端共享jar，通过接口的方式相互依赖，强依赖关系，需要严格的版本控制，对程序无入侵

OpenFeign

  • 消费端和提供端可以不用共享JAR，无接口依赖，定义好相关的json字段即可，对程序有一定入侵性
  • 程序与文本之间转换效率低，不适合内部大数据交换
  • 相比于二进制RPC，传输需要的带宽更大

API网关

Posted on 2020-06-09 Edited on 2020-06-30 In 微服务

职能

  • 请求接入
  • 业务聚合
    • 下游服务注册及管理
    • API聚合与数据编排，针对API的定制化、复用
  • 中介策略。实现鉴权、反爬、路由、分流、流控、降级等策略
    • 基础安全保障，侧重应用层安保能力建设
    • 流量治理，侧重于流量隔离与调度
  • 统一管理。提供配置管理平台，对所有API服务的调用生命周期和中介策略进行统一管理

分类

  • 流量网关。与后端业务无关，侧重于全局流量的安全与稳定
  • 业务网关。针对某个业务的流控，下流多API聚合等

定位

作为网关，最重要的是安全稳定，其次是降本增效，如何在保证安全稳定的前提下，以最快的速度帮助业务达成，这就是API网关的定位

实现

Kong

Kong是一个在Nginx运行的Lua应用程序，由lua-nginx-module实现。Kong和OpenResty一起打包发行，其中已经包含了lua-nginx-module。OpenResty不是Nginx的分支，而是一组扩展其功能的模块。

Tyk

Tyk是一个开源的、轻量级的、快速可伸缩的 API网关，支持配额和速度限制，支持认证和数据分析，支持多用户多组织，提供全 RESTful API。基于 go 编写。

Traefik

Traefik 是一个现代 HTTP 反向代理和负载均衡器，可以轻松部署微服务，Traeffik 可以与您现有的组件（Docker、Swarm，Kubernetes，Marathon，Consul，Etcd，…）集成，并自动动态配置。

Ambassador

Ambassador 是一个开源的微服务 API 网关，建立在 Envoy 代理之上，为用户的多个团队快速发布，监控和更新提供支持，支持处理 Kubernetes ingress controller 和负载均衡等功能，可以与 Istio 无缝集成。

Netflix Zuul

Zuul是一种提供动态路由、监视、弹性、安全性等功能的边缘服务，是 Netflix 出品的一个基于JVM 路由和服务端的负载均衡器。

Zuul is an L7 application gateway that provides capabilities for dynamic routing, monitoring, resiliency, security, and more.

架构

主要分为三个模块，Filter管理、Filter加载、Filter运行时模块

  • 类型Type（pre、route、post、error）
  • 执行顺序Order
  • 条件Criteria
  • 动作Action

生命周期

网关优化

  • AsyncServlet优化连接数
  • Hystrix熔断限流（信号量隔离）
  • 连接池管理，设置上线
  • CAT和Hystrix监控

Zuul2.0

Zuul 1是阻塞多线程模式，适用于计算密集型场景

  • 编程模型简单、开发调试运维简单、容易监控迈点、十亿级OK
  • 线程上下文切换开销、连接数限制、延迟阻塞耗尽线程连接资源

Zuul 2是非阻塞异步模式,适用于IO密集型场景

  • 线程开销少、连接数易扩展、可支撑百/千亿级的量
  • 线程模型复杂，开发调试运维复杂、ThreadLocal不work

SpringCloud Zuul

  • 去掉了一些生产级特性，如动态过滤器加载，小规模项目可以使用
  • 注重开发者的体验
  • 启动时批量加载Filter

SpringCloud Gateway

Gateway is built on Spring Framework 5, Project Reactor and Spring Boot 2 using non-blocking APIs. Websockets are supported and it’s a much better developer experience since it’s tightly integrated with Spring.

  • 客户端向 Spring Cloud Gateway 发出请求。
  • 然后在 Gateway Handler Mapping 中找到与请求相匹配的路由，将其发送到 Gateway Web Handler。
  • Handler再通过指定的过滤器链来将请求发送到我们实际的服务执行业务逻辑，然后返回。
  • 过滤器之间用虚线分开是因为过滤器可能会在发送代理请求之前（“pre”）或之后（“post”）执行业务逻辑。

微服务架构

Posted on 2020-06-08 Edited on 2020-06-30 In 微服务

推荐阅读

the microservice architectural style is an approach to developing a single application as a suite of small services, each running in its own process and communicating with lightweight mechanisms, often an HTTP resource API.

These services are built around business capabilities and independently deployable by fully automated deployment machinery. There is a bare minimum of centralized management of these services, which may be written in different programming languages and use different data storage technologies.

定义

  • 一种架构风格，将单体应用划分成一组⼩小的服务，服务之间相互协作，实现业务功能

  • 每个服务运⾏在独立的进程中，服务间采用轻量级的通信机制协作(通常是HTTP/ JSON)

  • 每个服务围绕业务能力进行构建，并且能够通过自动化机制独立地部署

  • 很少有集中式的服务管理，每个服务可以使用不同的语言开发，使⽤不同的存储技术

  • 基于有界上下文的，松散耦合的面向服务的架构

微服务的权衡

优点

  • 强大的模块边界：微服务加强了模块化结构，这对于大型团队而言尤其重要。
  • 独立部署：简单服务更易于部署，并且由于它们是自治的，因此出错时不太可能导致系统故障。
  • 技术多样性：使用微服务，您可以混合使用多种语言，开发框架和数据存储技术。

缺点

  • 分发：分布式系统更难编程，因为远程调用速度很慢并且总是有失败的风险。
  • 最终一致性：对于分布式系统而言，保持强一致性非常困难，这意味着每个人都必须管理最终一致性。
  • 运营复杂性：您需要一个成熟的运营团队来管理大量定期重新部署的服务。

微服务设计要素

服务fail设计

  • 对应用程序进行设计，以便它们可以容忍服务故障，若目标服务不可用，必须尽可能优雅地做出响应
  • 由于服务随时可能发生故障，微服务要注重应用程序的实时监控，检查架构元素（数据库每秒收到多少请求）和业务相关指标（例如每分钟收到多少订单）
  • 可以看到每个服务的复杂监视和日志记录设置，例如显示上/下状态的仪表板以及各种与业务和业务相关的指标。断路器状态，当前吞吐量和等待时间的详细信息。

服务优化设计

  • 我们经常需要对服务进行优化，将服务分解作为进一步的工具，使用正确的态度和工具，可以频繁、快速
  • 组件的关键特性是独立替换和可升级性的概念，将应用程序分解为组件、重写组件时需要设想是否不影响其他collaborators
  • 实际上许多微服务的规划是短暂（会被废弃）而不是长久的
  • 如果两个服务经常需要一起进行修改，说明说它们应该被合并
  • 微服务部署时一项服务的更改会对其他服务造成影响，传统的集成方法是通过版本进行控制，微服务的首选仅是把版本控制作为最后手段

思考？

  • 很难弄清楚组件的边界应该在哪里
  • 如果组件组成不清晰，那我们需要将复杂性从组件内部转移到组件之间的连接
  • 从整体开始，使其保持模块化，并在整体出现问题后将其拆分为微服务
